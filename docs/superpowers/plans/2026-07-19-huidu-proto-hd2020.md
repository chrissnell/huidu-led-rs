# `huidu-proto` HD2020 Gen6 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add the HD2020 Gen6 realtime protocol to `huidu-proto` — its own frame envelope and command set, a monochrome bitmap encoder for text panels, a bundled bitmap font blitted by hand, the `ProtoError::Hd2020` variant, and golden fixtures — all I/O-free, cleanly isolated from SDK 2.0 under `huidu-proto::hd2020`.

**Architecture:** HD2020 Gen6 is a *separate* wire protocol that shares only the TCP port with SDK 2.0 (`DESIGN.md §6`). It does not reuse the `<len:u16le><cmd:u16le>` envelope; instead the host rasterizes content to a 1-bpp bitmap and pushes it straight to the panel. The subsystem is three modules under `crates/huidu-proto/src/hd2020/`:

- `frame.rs` — the Gen6 frame (`START | cmd | len:u16le | payload | checksum`) and the `Hd2020Cmd` command set, with `encode`/`decode`.
- `bitmap.rs` — `MonoBitmap` (packed 1-bpp, MSB-leftmost), a hand-written glyph blit, `render`/`measure` for laying text into a panel, and `to_payload` (the on-wire `[w:u16le][h:u16le][rows…]`).
- `font.rs` — a bundled 5×7 bitmap font embedded with `include_bytes!("font5x7.bin")`; no TTF, no `ab_glyph`.

**Tech Stack:** Rust (edition 2021), `bytes`, `thiserror`. No new crate dependencies. Font asset generated by `tools/gen_hd2020_font.py`.

**Scope note:** This is subsystem **3 of 8** from `DESIGN.md §12`. It depends only on subsystem 1 (core — merged, PR #1): it adds a `ProtoError` variant and a new top-level module, and reuses nothing from the raw SDK codec by design. It does *not* touch the `huidu` client crate — protocol auto-detection and HD2020 command routing are subsystem 8 (`GRA-399`), which consumes this encoder.

## Wire-format decision to lock in (and verify)

`DESIGN.md §6` describes HD2020 Gen6 only in prose: a separate framing, bitmap encoding, and a bundled Plan 9 font. It does **not** give the Gen6 start byte, command numbering, length span, checksum, or bitmap header, and **no Go reference (`hd2020_gen6_test.go`) or hardware capture is available in this workspace.** This plan therefore fixes a self-consistent, round-trippable placeholder so the encoder, decoder, bitmap layer, and fixtures can be built and tested now:

- **Frame:** `0xA5` start byte, 1-byte command, `u16le` payload length, payload, 1-byte additive checksum over `cmd + len + payload`.
- **Commands:** `RealtimeText = 0x01`, `Clear = 0x02`, `Brightness = 0x03`.
- **Bitmap payload:** `[width:u16le][height:u16le][packed 1-bpp rows]`, stride `ceil(width/8)`, MSB = leftmost pixel.
- **Font:** hand-authored 5×7 (not the Plan 9 7×13 tables yet).

When the Go reference or a real capture lands, only the constants in `frame.rs`/`bitmap.rs`, the font asset, and the fixture bytes change — the module shape and public API do not. This mirrors how core locked its `<len>` span. Every consumer reads metrics through the `font.rs` constants, so a Plan 9 7×13 swap is a drop-in (see `src/hd2020/FONT-LICENSE.md`).
